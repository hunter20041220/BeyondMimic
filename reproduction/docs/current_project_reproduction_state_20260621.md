## What Is Still Missing For Paper-Level Virtual Reproduction

Excluding real robot deployment, the remaining high-priority missing items are:

1. Clean official unpatched G1 conversion/replay

   Need a paper-facing `csv_to_npz.py` / `replay_npz.py` entry path that does not depend on local capture/workaround language. Current loop-body and importer-export evidence is strong but still not the same as a clean official entry success.

2. Paper-level tracking teacher quality

   Need a much stronger PPO tracking policy with lower falls/terminations and reward/termination components close enough to justify using it as a teacher. The first robot-order FK PPO multiseed eval is complete and stable, but a diagnostic shows a deterministic step-0 reset/bootstrap spike plus persistent post-step0 done rate, so reset/target alignment and `ee_body_pos` termination must be inspected before downstream data collection.

3. True DAgger rollout dataset

   Need accepted teacher/student rollouts with provenance, perturbation protocol, accept/reject records, coverage, split discipline, and state/action/latent traces. Current rollout datasets are local virtual teacher candidates, not official DAgger logs.

4. VAE closed-loop control from a strong teacher

   Current VAE reconstruction and local closed-loop evaluations are useful, but the source teacher is weak/local. A paper-facing VAE result needs a stronger tracking teacher and a clearer DAgger dataset.

5. Full diffusion Transformer training/evaluation on teacher-rollout state-latent trajectories

   The code path exists and local denoisers have trained, but there is no official checkpoint or strong paper-level dataset. Need stronger data and closed-loop evaluation.

6. Fig. 5 / Fig. 6 paper-level virtual experiments

   Need protocol-aligned joystick, waypoint, walk-to-run, inpainting, obstacle avoidance, and composed guidance rollouts, with success/fall/collision metrics and videos. Current assets are local proxy evidence.

7. TensorRT/asynchronous deployment audit

   Need TensorRT or at least CUDA-provider runtime, latency, engine artifacts, async scheduling, and paper-like deployment boundary. Current CPU ONNXRuntime audit is not enough.

8. Completion matrix and goal text cleanup

   Some old text still reflects earlier states. For example, PPO smoke was previously listed as blocked even though current local PPO training/eval exists. This round updates that row, but the new goal should explicitly separate local virtual milestones from paper-level gates.

9. Tracking reset and termination-quality gate

   The latest robot-order FK PPO diagnostic shows `2048/2048` environments terminating at step 0 in all three eval seeds, with step-0 body-position error around `43.29` m. Excluding step 0 reduces mean body-position error from about `0.360` to about `0.216`, but the remaining done rate is still about `0.176`. This points to reset/bootstrap alignment and `ee_body_pos` termination behavior as the immediate blocker before stronger PPO or downstream model reruns.
